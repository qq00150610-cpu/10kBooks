## 1. 整体架构设计

### 1.1 设计原则

万卷书苑平台采用**微服务架构**设计，遵循以下核心原则：

| 原则 | 说明 |
|------|------|
| **高可用性** | 服务冗余部署，自动故障转移，保证 99.9% 以上可用性 |
| **可扩展性** | 水平扩展能力，支持按需扩容，应对流量高峰 |
| **松耦合** | 服务间通过 API 通信，降低依赖，提高独立部署能力 |
| **可观测性** | 全链路追踪、监控告警、日志集中管理 |
| **安全性** | 多层防护，数据加密，身份认证与授权 |
| **容错性** | 服务降级、熔断、重试等机制，保证系统稳定性 |

### 1.3 核心服务域划分

| 服务域 | 包含服务 | 职责说明 |

| **社交域** | 消息服务、通知服务、动态服务、关系服务 | 即时通讯、系统通知、信息流推送、社交关系 |
| **搜索域** | 搜索服务、推荐服务、标签服务 | 全文检索、智能推荐、内容分类标签 |
| **AI 域** | 推荐服务、翻译服务、NLP服务 | 机器学习推荐、多语言翻译、自然语言处理 |

## 4. 数据流向

### 4.1 用户请求处理流程

```
┌─────────────────────────────────────────────────────────────────────────┐
│                           用户请求处理流程                               │
└─────────────────────────────────────────────────────────────────────────┘

  [用户操作]
       │
       ▼
  ┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐
  │  客户端   │────▶│  CDN    │────▶│  网关   │────▶│  服务   │────▶│  数据库  │
  │  (App)   │     │  缓存   │     │ Gateway │     │ Service │     │ Storage │
  └─────────┘     └─────────┘     └────┬────┘     └────┬────┘     └─────────┘
                                      │               │
                               ┌──────▼──────┐ ┌──────▼──────┐
                               │   限流      │ │   认证      │
                               │  RateLimit  │ │  Auth       │
                               └─────────────┘ └─────────────┘

  响应流程：数据库 ──▶ 服务 ──▶ 网关 ──▶ 客户端
```

### 4.2 核心数据流向

#### 4.2.1 内容发布流程

```
作者创作 ──▶ 内容服务 ──▶ 审核服务 ──▶ 发布 ──▶ 搜索引擎索引 ──▶ 推荐系统
                │                                        │
                ▼                                        ▼
           MongoDB (原始内容)                    MySQL (已发布内容)
```

#### 4.2.2 用户下单流程

```
用户下单 ──▶ 订单服务 ──▶ 支付网关 ──▶ 第三方支付 ──▶ 支付回调
                │                                        │
                ▼                                        ▼
          MySQL (订单)                            钱包服务 (余额)
                                                     │
                                                     ▼
                                              通知服务 (推送通知)
```

#### 4.2.3 消息推送流程

```
用户发送消息 ──▶ 消息服务 ──▶ Kafka ──▶ 消息处理 ──▶ Redis 缓存 ──▶ WebSocket ──▶ 目标用户
                        │                                              │
                        ▼                                              ▼
                   消息存储 (MySQL)                              离线消息存储
```

### 4.3 缓存策略

| 数据类型 | 缓存策略 | 过期时间 | 说明 |
|---------|---------|---------|------|
| 用户 Session | Redis String | 7 天 | 用户登录状态 |
| 用户信息 | Redis Hash | 1 小时 | 用户基本信息 |
| 书籍列表 | Redis List | 5 分钟 | 首页推荐书籍 |
| 书籍详情 | Redis Hash | 30 分钟 | 书籍详细信息 |
| 章节内容 | 本地缓存 + Redis | 24 小时 | 章节阅读内容 |
| 排行榜 | Redis ZSet | 1 小时 | 各类榜单数据 |
| 搜索结果 | Elasticsearch | - | 搜索缓存 |

### 4.4 异步处理

关键异步处理场景：

1. **消息推送**：用户触发 → Kafka → 消息服务 → WebSocket/推送
2. **内容审核**：发布内容 → Kafka → 审核服务 → 结果通知
3. **积分变动**：交易完成 → Kafka → 积分服务 → 用户积分更新
4. **日志收集**：业务日志 → Kafka → Logstash → Elasticsearch
5. **数据统计**：用户行为 → Kafka → 统计服务 → 数据仓库

## 6. 扩展性设计

### 6.1 水平扩展策略

| 组件 | 扩展方式 | 扩展上限 | 说明 |
|------|---------|---------|------|
| **API 网关** | 多实例 + 负载均衡 | 无限制 | 无状态，可快速扩容 |
| **业务服务** | K8s HPA 自动扩缩容 | 按需 | 根据 CPU/内存/自定义指标 |
| **MySQL** | 主从复制 + 分库分表 | 1024 库 | 业务拆分 + MySQL Cluster |
| **Redis** | Cluster 模式 | 16384 槽 | 自动分片 + 副本 |
| **Elasticsearch** | 集群分片 | 按需 | 主分片 + 副本分片 |
| **Kafka** | 分区扩展 | 数千分区 | 分区并行消费 |

### 6.2 服务注册与发现

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         服务注册与发现流程                               │
└─────────────────────────────────────────────────────────────────────────┘

  服务启动时：                                          服务调用时：
  ┌─────────┐                                          ┌─────────┐
  │ 服务实例 │ ──▶ 注册服务信息 ──▶ ┌─────────┐         │ 调用方   │
  └─────────┘                       │  Nacos  │◀──────── │         │
                                   │ 注册中心  │  查询   └────┬────┘
                                   └────┬─────┘   服务列表     │
                                        │                       │
                                   心跳保活                    ▼
                                        │                ┌─────────┐
                                   ┌────▼─────┐         │ 被调用方 │
                                   │ 服务实例  │         │ (负载均衡)│
                                   └──────────┘         └─────────┘
```

### 6.3 负载均衡策略

| 策略 | 适用场景 | 配置方式 |
|------|---------|---------|
| **轮询 (Round Robin)** | 无状态服务 | 默认策略 |
| **加权轮询 (Weighted)** | 异构环境 | 根据实例配置权重 |
| **最少连接 (Least Connections)** | 长连接服务 | IM、推送服务 |
| **IP 哈希 (IP Hash)** | 会话保持 | 用户会话一致性 |
| **响应时间 (Response Time)** | 性能敏感 | 动态调整 |

### 6.4 容错机制

| 机制 | 说明 | 配置 |
|------|------|------|
| **超时重试** | 请求超时自动重试 | 默认 3 次，指数退避 |
| **熔断器** | 故障时快速失败 | Sentinel 熔断策略 |
| **限流** | 防止雪崩 | 按接口/用户限流 |
| **降级** | 核心功能优先 | 降级非核心功能 |
| **隔离** | 资源隔离 | 线程池隔离、服务隔离 |
